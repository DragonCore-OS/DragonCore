# Meeting Protocol + 4h Endurance 聯合驗證計劃

**版本**: v0.1  
**日期**: 2026-03-20  
**目標**: 驗證會議協議層上線後，系統能否保持長時穩定、可回放、可收斂

---

## 驗證目標

回答核心問題：

> **會議協議層上線後，DragonCore 是否仍能保持 4 小時長時穩定、可回放、可收斂？**

這比繼續添加新功能更有價值，因為：
1. 會議協議層是新增複雜度最高的組件
2. 長時運行會暴露 stance 收斂、事件堆積、內存增長等問題
3. 只有聯合驗證通過，才能確認會議層的「生產就緒」狀態

---

## 驗證設計

### 時間與規模

| 參數 | 設定 |
|------|------|
| 總運行時間 | 4 小時 (240 分鐘) |
| 會議輪數 | 20-30 輪 |
| 每輪會議 | 3-4 個階段轉換 |
| Seat 參與 | 19 席全部參與 |
| Provider | 雙 provider (GPT-OSS-120B + Kimi CLI) |
| 人工干預 | 0 |

### 會議場景設計

每輪會議包含完整流程：

```
1. Open      - 開啟會議，設定議題
2. Assemble  - 組裝會議
3. RollCall  - 點名確認
4. TopicLock - 鎖定議題
5. OpenFloor - 開放發言（3-5 輪發言）
   - Request-speak 隊列
   - SmartModerator 調度
   - Stance 更新
6. ChallengeWindow - 挑戰窗口
7. DraftResolution - 決議草案
8. CommitAction    - 提交行動
9. Closed          - 會議關閉
```

### 議題設計

| 議題類型 | 頻率 | 目的 |
|----------|------|------|
| 戰略規劃 | 30% | 測試 Strategic/Conservative 行為 |
| 風險評估 | 30% | 測試 VerificationFirst/Aggressive 行為 |
| 資源分配 | 25% | 測試 ResourceBound 行為 |
| 衝突決議 | 15% | 測試 stance 收斂、challenge 機制 |

## 驗證步驟

### Phase 1: 預驗證 (30 分鐘)

1. 編譯 release 版本
2. 運行單元測試 (79 項全部通過)
3. 運行短時會議測試 (1 輪完整會議)
4. 驗證 replay 功能正常

### Phase 2: 主驗證 (4 小時)

1. 啟動 DragonCore runtime
2. 啟動自動化會議生成器
3. 監控運行狀態（無人工干預）
4. 記錄所有指標

### Phase 3: 後驗證 (30 分鐘)

1. 運行 replay 驗證
2. 檢查所有指標
3. 生成驗證報告

**結論**: 需要修復問題後重新驗證。
